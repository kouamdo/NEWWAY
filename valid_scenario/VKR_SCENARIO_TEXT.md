# Валидированный сценарий bidirectional coupling ns-3 <-> SUMO (готовый фрагмент для ВКР)

## Постановка задачи

Для верификации двунаправленного сопряжения транспортного симулятора SUMO и сетевого симулятора ns-3 сформирован сценарий, в котором сетевые потери напрямую влияют на поведение транспортных агентов, а транспортные события (инцидент/столкновение/затор) отражаются в сетевых логах и метриках.

Сценарий реализован как воспроизводимый пакет `valid_scenario` и запускается одной командой `valid_scenario/run.sh`.

## Логика эксперимента

В правой полосе движется колонна из четырех транспортных средств: `veh2` (лидирующее, инцидентное), `veh3`, `veh4`, `veh5`.

- В момент `t = 6 s` для `veh2` инициируется инцидент (остановка на полосе).
- `veh3` и `veh5` имеют нормальную связь и получают предупреждения (CAM/CPM), что приводит к перестроению в безопасную полосу.
- Для `veh4` включен таргетированный профиль потерь PHY-уровня (`CAM/CPM = 1.0`), поэтому предупреждения систематически теряются.
- В strict-режиме (`drop-triggered-reaction-enable=0`) потери не порождают обходного «реактивного» маневра: для каждого drop фиксируется `drop_decision_no_action`.
- Следствием является столкновение `veh4` с `veh2`.
- Параметры `collision.action=warn` и `collision.stoptime=1000` удерживают столкнувшиеся авто на дороге, формируя устойчивое место ДТП и затор.

## Почему это именно bidirectional coupling

1. **SUMO -> ns-3:** транспортный инцидент (остановка `veh2`) создает опасную ситуацию, которая порождает поток предупреждающих сообщений и событий в ns-3.
2. **ns-3 -> SUMO:** таргетированные сетевые потери для `veh4` приводят к отсутствию маневра (`drop_decision_no_action`) и к транспортному исходу (столкновение в SUMO).

Таким образом, реализована причинно-следственная петля между двумя симуляторами, а не односторонняя визуализация.

## Доказательная база

Сценарий формирует согласованные артефакты:

- `eva-collision.xml`: факт столкновения (время, участники, полоса, скорость).
- `drop_decision_timeline/event_timeline.csv`: ID-aware цепочка `pkt_uid: DROP_PHY -> DECISION`.
- `collision_causality/collision_causality.csv`: автоматический causal-audit `loss -> no_action -> collision`.
- `valid_scenario_story/*.png`: объединенные графики поведения в SUMO и событий в ns-3.

Ключевое преимущество доказательства состоит в связи уровня отдельного пакета (`pkt_uid`) с итоговым транспортным событием.

## Интерпретация графиков для раздела результатов

1. `speed_lane_timeseries.png`
- показывает временную эволюцию скоростей и полос для `veh2..veh5`;
- демонстрирует раннее перестроение `veh3` и более позднее перестроение `veh5`.

2. `gap_ttc_timeseries.png`
- показывает сближение и деградацию safety-прокси (`gap`, `TTC`) перед столкновением;
- вертикальная линия collision-time позволяет связать изменение прокси с фактом ДТП.

3. `ns3_events_per_second.png`
- показывает рост потока `DROP_PHY` и `drop_decision_no_action` у `veh4`;
- одновременно отображает моменты lane-change решений `veh3`/`veh5`.

4. `event_chain_timeline.png` и `event_chain.csv`
- дают компактную причинную шкалу: `incident -> veh3 lane change -> veh4 collision -> veh5 lane change`.

## Воспроизводимость

Сценарий упакован в отдельный каталог `valid_scenario`.

Базовый запуск:

```bash
valid_scenario/run.sh
```

Live-демонстрация с SUMO GUI:

```bash
SUMO_GUI=1 valid_scenario/run.sh
```

Все выходные данные формируются в едином run-каталоге, что позволяет приложить их в раздел «Результаты экспериментов» без ручной постобработки.
